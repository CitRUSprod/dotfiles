---
description: Применяет исправления из комментариев PR с отслеживанием уже применённых замечаний через state-файл
---

Ничего не коммитить и не добавлять в индекс (git add) — только редактировать файлы.

1.  **Проверь окружение:**
    - Удали старые временные файлы от предыдущих запусков (если остались после ошибки):
        ```
        find "${TMPDIR:-/tmp}" -maxdepth 1 -name 'opencode-prfix-*' -mmin +60 -delete 2>/dev/null || true
        ```
    - Проверь, что текущий репозиторий — chezmoi source directory (наличие `.chezmoiignore` в корне). Если нет — сообщи об ошибке и прекрати выполнение.
    - Проверь, доступен ли `gh` CLI (`gh --version`). Если недоступен — сообщи об ошибке и прекрати выполнение.
    - Проверь авторизацию `gh` (`gh auth status`). Если не авторизован — сообщи об ошибке и предложи выполнить `gh auth login`, затем прекрати выполнение.

2.  **Определи PR:**
    - Проанализируй `$ARGUMENTS` (если передан и не пуст):
        - Если это число (только цифры) — используй как номер PR (`pr_number`).
        - Если это URL вида `https://github.com/.../pull/N` — извлеки номер PR.
        - Если в тексте упоминается номер PR или URL — извлеки их.
        - Если не удалось извлечь PR — значит, аргументы содержат только пожелания по критичности; определи PR для текущей ветки.
    - Если PR ещё не определён — получи для текущей ветки: `gh pr view --json number,url,headRefName,baseRefName,title,headRefOid`
    - Из результата извлеки: `pr_number`, `pr_url`, `headRefOid` (SHA последнего коммита в PR).
    - Если PR не найден или команда завершилась ошибкой — сообщи пользователю и прекрати выполнение.

3.  **Определи фильтр критичности:**
    - По умолчанию: применяй замечания с критичностью 🔴 (Критическая), 🟠 (Высокая), 🟡 (Средняя). ⚪ (Низкая) игнорируется.
    - Проанализируй `$ARGUMENTS`: если пользователь упоминает "все", "включая низкие", "include low", "low", "низк" или иным образом просит не фильтровать — применяй замечания любой критичности (включая ⚪).

4.  **Получи state-файл:**
    - Проверь наличие `.pr-fix-state.json` в корне репозитория.
    - Если файл существует — загрузи список `applied_comment_ids` (массив ID уже применённых комментариев).
    - Если файла нет — инициализируй пустой список.

5.  **Получи комментарии PR:**
    - Определи owner и repo из `pr_url` (`https://github.com/{owner}/{repo}/pull/{number}`).
    - Получи issue-комментарии (общие обсуждения PR):
        ```
        gh api /repos/{owner}/{repo}/issues/{pr_number}/comments --jq '.[] | {id, body, created_at, user: .user.login}'
        ```
    - Получи review-комментарии (привязанные к строкам кода):
        ```
        gh api /repos/{owner}/{repo}/pulls/{pr_number}/comments --jq '.[] | {id, body, path, original_line, commit_id, created_at, user: .user.login}'
        ```
    - Объедини оба списка.
    - Отфильтруй: исключи комментарии, чей `id` уже есть в `applied_comment_ids`.
    - Если после фильтрации комментариев не осталось — сообщи "Нет неприменённых комментариев" и прекрати выполнение.

6.  **Извлеки и отфильтруй замечания:**
    - Для каждого комментария:
        - Найди в `body` блоки, начинающиеся с маркера `<!-- pr-fix:` и эмодзи критичности (например, `<!-- pr-fix: 🔴 -->`). Если маркер отсутствует — ищи блоки, начинающиеся с эмодзи критичности (🔴 🟠 🟡 ⚪) как fallback.
        - Каждый блок обычно содержит секции:
            - `**Файл и строка:**` — путь к файлу и номер строки (может отсутствовать)
            - `**Суть проблемы:**` — описание
            - `**Рекомендация:**` — что нужно сделать
        - Для review-комментариев (есть `path` и `original_line`) используй `original_line` как более точный источник строки (устойчив к сдвигам при новых коммитах).
    - Собери все найденные замечания в единый список.
    - Отфильтруй по критичности согласно шагу 3.
    - Если после фильтрации замечаний не осталось — сообщи "Все замечания отфильтрованы (остались только ⚪)" и прекрати выполнение.

7.  **Примени исправления:**
    - Для каждого замечания (сортируй по критичности: 🔴 → 🟠 → 🟡 → ⚪):
        - Если указаны файл и строка (из `**Файл и строка:**` или из `path`/`original_line` review-комментария):
            - Прочитай соответствующий файл.
            - Найди указанную строку (или диапазон строк).
            - Примени рекомендацию из секции `**Рекомендация:**`.
        - Если файл или строка не указаны:
            - Проанализируй рекомендацию и описание проблемы.
            - Примени исправление глобально: поиском по проекту, рефакторингом или другими подходящими изменениями.
    - Редактируй исходники в `~/.local/share/chezmoi/` (не целевые файлы в `~`).
    - После применения замечания — добавь `id` комментария в `applied_comment_ids`.
    - Если при применении возникли конфликты или ошибки — запиши их и продолжай с остальными замечаниями.

8.  **Сообщи результат:**
    - Сохрани `.pr-fix-state.json` с обновлённым `applied_comment_ids`.
    - Выполни `chezmoi apply -v` для синхронизации исходников с `~`.
    - Перечисли применённые исправления: критичность, файл, строка, суть.
    - Укажи пропущенные (⚪, если игнорировались).
    - Укажи, какие замечания не удалось применить и почему.
    - Выведи ссылку на PR: `pr_url`.
