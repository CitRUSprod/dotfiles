---
description: Применяет исправления из комментариев PR
---

Ничего не коммитить и не добавлять в индекс (git add) — только редактировать файлы.

1.  **Проверь окружение:**
    - Проверь, что текущая директория — git-репозиторий (`git rev-parse --git-dir`). Если нет — сообщи об ошибке и прекрати выполнение.
    - Проверь, доступен ли `gh` CLI (`gh --version`). Если недоступен — сообщи об ошибке и прекрати выполнение.
    - Проверь авторизацию `gh` (`gh auth status`). Если не авторизован — сообщи об ошибке и предложи выполнить `gh auth login`, затем прекрати выполнение.
    - Проверь, что нет незакоммиченных изменений: `git diff --quiet`. Если есть — предупреди пользователя и запроси подтверждение на продолжение.

2.  **Определи PR:**
    - Проанализируй `$ARGUMENTS` (если передан и не пуст):
        - Если это число (только цифры) — используй как номер PR (`pr_number`).
        - Если это URL вида `https://github.com/.../pull/N` — извлеки номер PR.
        - Если в тексте упоминается номер PR или URL — извлеки их.
        - Если не удалось извлечь PR — выведи предупреждение "Аргументы не распознаны как номер или URL PR. Использую PR для текущей ветки." и определи PR для текущей ветки.
    - Если PR ещё не определён — получи для текущей ветки: `gh pr view --json number,url,headRefName,baseRefName,title,state`
    - Из результата извлеки: `pr_number`, `pr_url`, `pr_state`.
    - Проверь, что PR не merged и не closed. Если `pr_state == "MERGED"` или `pr_state == "CLOSED"` — сообщи "PR #{pr_number} {pr_url} уже {merged/closed}. Применение исправлений к закрытому PR бессмысленно." и прекрати выполнение.
    - Если PR не найден или команда завершилась ошибкой — сообщи пользователю и прекрати выполнение.
    - Валидируй извлечённый `pr_number` — только цифры. При несоответствии — прекрати выполнение с сообщением об ошибке.

3.  **Определи фильтр критичности:**
    - По умолчанию: применяй замечания с критичностью 🔴 (Критическая), 🟠 (Высокая), 🟡 (Средняя). ⚪ (Низкая) игнорируется.
    - Проанализируй `$ARGUMENTS`: если пользователь упоминает "все", "включая низкие", "include low", "low", "низк" или иным образом просит не фильтровать — применяй замечания любой критичности (включая ⚪).

4.  **Получи комментарии PR:**
    - Определи owner и repo из `pr_url` (`https://github.com/{owner}/{repo}/pull/{number}`). Валидируй: `owner` и `repo` — только буквы, цифры, дефисы. При несоответствии — прекрати выполнение с сообщением об ошибке.
    - Получи issue-комментарии (общие обсуждения PR):
        ```
        gh api /repos/{owner}/{repo}/issues/{pr_number}/comments --paginate --jq '.[] | {id, body, created_at, user: .user.login}'
        ```
    - Получи review-комментарии (привязанные к строкам кода):
        ```
        gh api /repos/{owner}/{repo}/pulls/{pr_number}/comments --paginate --jq '.[] | {id, body, path, line, original_line, commit_id, created_at, user: .user.login}'
        ```
    - Получи review-summary комментарии (общие замечания по всему PR):
        ```
        gh api /repos/{owner}/{repo}/pulls/{pr_number}/reviews --paginate --jq '.[] | select(.state == "CHANGES_REQUESTED") | {id, body, user: .user.login}'
        ```
    - Проверь код возврата каждого запроса `gh api`. При ошибке HTTP 403/429 (rate limiting) — выполни экспоненциальный backoff: подожди 2^N секунд (N — номер попытки, до 3 попыток) и повтори запрос. Если после 3 попыток ошибка сохраняется — сообщи пользователю о превышении лимита GitHub API и прекрати выполнение. При других ошибках — сообщи пользователю и прекрати выполнение.
    - Объедини все полученные комментарии в один список.
    - Дедуплицируй комментарии по композитному идентификатору (`issue_<id>`, `review_<id>`, `summary_<id>`), чтобы избежать коллизий между разными типами комментариев (их id — из разных пространств GitHub API).
    - Если комментариев нет — сообщи "Нет комментариев в PR" и прекрати выполнение.

5.  **Извлеки и отфильтруй замечания:**
    - Веди список `applied_comment_ids` в памяти — композитные идентификаторы уже обработанных комментариев (с префиксом источника, как на шаге 4). Инициализируй пустым.
    - Для каждого комментария:
        - Пропусти комментарий, если его композитный идентификатор (с префиксом источника) уже есть в `applied_comment_ids`.
        - Найди в `body` блоки замечаний. Блок — это текст, начинающийся с новой строки, содержащей эмодзи критичности (🔴 🟠 🟡 ⚪) или их текстовые аналоги (Критическая, Высокая, Средняя, Низкая, CRITICAL, HIGH, MEDIUM, LOW) в начале строки. Игнорируй эмодзи внутри markdown-блоков кода (```...```) и цитат (> ...).
        - Каждый блок содержит секции:
            - `**Файл и строка:**` — путь к файлу и номер строки (может отсутствовать)
            - `**Суть проблемы:**` — описание
            - `**Рекомендация:**` — что нужно сделать
        - Если после парсинга body не содержит ни одного блока с критичностью — пропусти комментарий и логируй предупреждение.
        - Для определения файла и строки используй приоритет:
            1. `path` и `line` из review-комментария (если `line` не `null`) — наивысший приоритет.
            2. `path` и `original_line` из review-комментария (если `line` — `null`).
            3. `**Файл и строка:**` из body — наинизший приоритет.
        - Предупреди пользователя, если используется `original_line` вместо `line` (комментарий может быть устаревшим).
    - Собери все найденные замечания в единый список.
    - Отфильтруй по критичности согласно шагу 3.
    - Если после фильтрации замечаний не осталось — сообщи "Все замечания отфильтрованы (остались только ⚪)" и прекрати выполнение.

6.  **Примени исправления:**
    - Для каждого замечания: группируй по файлу, затем сортируй внутри файла от конца к началу (снизу вверх по номеру строки). Критичность используй только как вторичный признак сортировки (при равной строке выше критичность).
    - После каждого изменения перечитывай файл заново, чтобы номера строк не сдвигались.
        - Если указаны файл и строка (из `**Файл и строка:**` или из `path`/`line`/`original_line` review-комментария):
            - Резолвь путь относительно корня репозитория (через `realpath -m "$(git rev-parse --show-toplevel)/$relative_path"` или `readlink -f` как fallback, если `realpath` недоступен). Проверь, что результат начинается с корня репозитория. Если нет — пропусти замечание и предупреди пользователя о path traversal.
            - Проверь, что файл существует. Если нет — сообщи пользователю и пропусти замечание.
            - Прочитай соответствующий файл.
            - Найди указанную строку (или диапазон строк).
            - Примени рекомендацию из секции `**Рекомендация:**`.
        - Если файл или строка не указаны:
            - Проанализируй рекомендацию и описание проблемы.
            - Примени исправление в пределах одного файла (того же, что упоминается в комментарии, или соседнего). Не делай глобальный рефакторинг — ограничь scope одним файлом.
    - После каждого успешно применённого замечания добавь его композитный идентификатор (с префиксом: `issue_<id>`, `review_<id>`, `summary_<id>`) в `applied_comment_ids`.
    - Если при применении возникли конфликты или ошибки — запиши их и продолжай с остальными замечаниями.

7.  **Сообщи результат:**
    - Покажи пользователю diff сделанных изменений: `git diff`.
    - Перечисли применённые исправления: критичность, файл, строка, суть.
    - Укажи пропущенные (⚪, если игнорировались).
    - Укажи, какие замечания не удалось применить и почему.
    - Выведи ссылку на PR: `pr_url`.
