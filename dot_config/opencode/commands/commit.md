---
description: Создаёт коммит и пушит его (build) или предлагает сообщение (plan), анализируя историю проекта. Файлы нужно предварительно проиндексировать через git add.
---

Сначала изучи стиль коммитов в проекте:

```
git log --oneline -20 --format="%s"
```

Затем посмотри текущие изменения:

```
git status
git diff --stat
```

Если есть изменения, сгенерируй сообщение коммита в том же стиле, что и последние коммиты. Сообщение должно быть кратким, использовать conventional commit format: `тип(scope): описание`. Опционально можно указать scope — область изменений (например, `feat(api):`, `fix(auth):`).

Допустимые типы:
- `feat` — Новая функциональность
- `fix` — Исправление ошибки
- `docs` — Изменения в документации
- `style` — Изменения, не влияющие на смысл кода (форматирование, отступы и т.д.)
- `refactor` — Изменение кода без исправления ошибок и добавления функциональности
- `perf` — Изменения, улучшающие производительность
- `test` — Добавление или исправление тестов
- `build` — Изменения в системе сборки или внешних зависимостях
- `ci` — Изменения в конфигурации CI
- `chore` — Прочие изменения, не затрагивающие src или тесты
- `revert` — Откат предыдущего коммита

Описание должно быть на том же языке, что и проект, и начинаться с заглавной буквы.

Примеры:
- `feat(api): Добавить эндпоинт для получения пользователей`
- `fix: Исправить ошибку валидации email`

Важно: команда не добавляет файлы в коммит автоматически. Перед вызовом /commit проиндексируйте нужные изменения через `git add <файлы>` или `git add -p`.

Если текущая сессия позволяет выполнять команды (build mode), выполни:

```
git commit -m "<сообщение>" && git push
```

Если сессия только для чтения (plan mode), просто выведи предложенное сообщение коммита без внесения изменений.
